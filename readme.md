# Brno Public Transport Departure Board

A minimalistic mobile-friendly departure board for public transport in Brno, Czechia. Enter a stop name to see upcoming departures, filter by specific routes, and bookmark your frequent stops.

Availabne here: https://brno-dpmb-online-timetable.onrender.com

## Architecture
The system operates using a three-part "Joiner" architecture to synchronize static schedules with live tracking:

1.  **WebSocket Collector (`collector.py`):** A background process that maintains a permanent connection to the Brno GIS StreamServer. it maintains a live JSON map of `vehicle_id` to `delay`. 
    *   *Example:* Vehicle `19100` has a `3.0` min delay.
2.  **GTFS-RT Binary Joiner (inside `app.py`):** The API periodically (30s cache) fetches the binary Protobuf feed from Kordis to create a cross-reference map between `trip_id` and vehicle `label`.
    *   *Example:* Trip `48109` is currently performed by vehicle `19100`.
3.  **API Service (`app.py`):** When a user requests a stop, the server queries SQLite for scheduled trips, finds the vehicle label via the Binary Joiner, and applies the specific delay from the Collector's JSON.
    *   *Final Result:* Trip `48109` (Schedule 10:00) + Delay (3 min) = **10:03**.
4. **Data Ingestor (`ingest.py`):** A pre-processing script that transforms raw GTFS text files into an optimized SQLite database.
    *   *Filtering:* Selects only Brno city stops (Zones 100, 101, 570) to reduce database size and improve query performance.
    *   *Normalization:* Standardizes time formats (e.g., padding `8:30` to `08:30:00`) and generates normalized stop names (stripping diacritics) for the search engine.
    *   *Indexing:* Creates relational indexes on `stop_id`, `trip_id`, and `departure_time` to ensure the API can filter millions of rows in milliseconds.

## Project Structure
```txt
.
├── data/
│   ├── gtfs/            # Unzipped GTFS .txt files (from gtfs.zip)
│   ├── live_delays.json # Data from WebSocket (delays)
│   └── transit.db       # SQLite database (generated by ingest.py)
├── src/
│   ├── static/          # Frontend assets (HTML, CSS, JS)
│   ├── ingest.py        # GTFS to SQLite ingestion script
│   ├── collector.py     # Background WebSocket listener
│   ├── app.py           # Flask Backend & RT Joiner
│   ├── start.sh         # Container startup script (runs both collector and app)
│   └── requirements.txt # Python dependencies
├── Dockerfile           # Docker build instructions
└── docker-compose.yml   # Docker Compose configuration
```

## Setup & Running

### Option 1: Running via Docker Compose (Recommended)
This handles environment variables, volumes, and multi-process execution automatically.
```bash
docker-compose up --build
```
Access at `http://localhost:5000`.

---

### Option 2: Running Locally
**Prerequisites:**
- Python 3.12.3+
- `pipenv`

**Windows Warning:**
This project uses `zoneinfo` for strict Czechia time. On **Windows**, you must install `tzdata`:
```bash
pipenv install tzdata
```

**Steps:**
1. Install dependencies:
   ```bash
   pipenv install
   ```
2. Ingest data (ensure `data/gtfs/` contains the unzipped files):
   ```bash
   cd src
   pipenv run python ingest.py
   ```
3. Run Collector (Background):
   ```bash
   pipenv run python collector.py &
   ```
4. Run Server:
   ```bash
   pipenv run python app.py
   ```

---

### Option 3: Running via Docker Standalone
1. Build: `docker build -t brno-transit .`
2. Run: `docker run -p 5000:5000 -v $(pwd)/data:/app/data brno-transit`
